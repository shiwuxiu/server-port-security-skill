# 安全事件响应 Runbook

> 本文档定义了安全事件的标准处理流程，确保 Claude Code 在发现问题后能给出全局性的后续指导。

---

## 安全 SLI（服务等级指示器）

**系统安全合规定义：**

```
安全 SLI = 0 公网暴露数据库 + 0 UFW穿透容器 + 0 弱口令 + 0 勒索特征
```

**合规判定：**

| 指标 | 合格标准 | 检测方法 |
|------|----------|----------|
| 公网暴露数据库 | 0 个 | `external_scan.sh` + `port_scan.sh` |
| UFW穿透容器 | 0 个 | `iptables -L DOCKER-USER -n` |
| 弱口令服务 | 0 个 | 无认证探测 |
| 勒索特征 | 0 个 | `read_me` 索引检测 |

**最终输出必须包含合规判定：**
```
🟢 PASS - 系统安全合规
🔴 FAIL - 发现 N 项违规，详见 findings[]
```

## 事件分类与响应矩阵

### 🔴 CRITICAL（立即响应）

#### 事件1：勒索攻击特征检测

**触发条件：** Elasticsearch 存在 `read_me`、`backup` 等可疑索引

**响应流程：**

```
┌─────────────────────────────────────────────────────────────┐
│  1. 立即隔离                                                 │
│     ├─ 断开公网访问（云安全组/防火墙）                         │
│     └─ 停止 ES 服务：docker stop elasticsearch               │
│                                                             │
│  2. 取证保全                                                 │
│     ├─ 截图可疑索引列表                                       │
│     ├─ 导出攻击者留言（read_me 索引内容）                      │
│     └─ 备份日志：journalctl -u docker > es_incident.log      │
│                                                             │
│  3. 数据评估                                                 │
│     ├─ 检查其他索引是否被删除                                 │
│     ├─ 评估备份可用性                                        │
│     └─ 如有备份，计算 RTO                                    │
│                                                             │
│  4. 恢复决策                                                 │
│     ├─ 有备份 → 从备份恢复                                   │
│     └─ 无备份 → 数据丢失，记录教训                           │
│                                                             │
│  5. 根因修复                                                 │
│     ├─ ES 绑定 127.0.0.1                                    │
│     ├─ 启用认证（xpack.security.enabled=true）               │
│     └─ 部署 es_monitor.sh 监控                              │
└─────────────────────────────────────────────────────────────┘
```

**Claude Code 输出示例：**
```
🔴 CRITICAL: 检测到勒索攻击特征

发现 ES 索引 "read_me"，疑似 Meow Attack。
已完成以下检查：
- [x] 端口已隔离（ES 已停止）
- [x] 取证截图已保存

⚠️ Runbook 指示后续操作：
1. 检查是否有可用备份
2. 如有备份，从备份恢复数据
3. 修复后启用 ES 认证
4. 部署监控脚本防止复发

是否需要我帮你检查备份状态？
```

#### 事件2：Docker Daemon 暴露（2375/2376）

**触发条件：** Docker Daemon API 可公网访问

**风险等级：** 极危 - 攻击者可完全控制服务器

**响应流程：**

```
┌─────────────────────────────────────────────────────────────┐
│  1. 紧急阻断                                                 │
│     ├─ iptables -I INPUT -p tcp --dport 2375 -j DROP        │
│     └─ 或：云安全组移除 2375 端口                            │
│                                                             │
│  2. 入侵评估                                                 │
│     ├─ 检查容器列表：docker ps -a                            │
│     ├─ 检查可疑镜像：docker images                           │
│     ├─ 检查近期操作：docker events --since 24h               │
│     └─ 检查 crontab 是否被植入后门                          │
│                                                             │
│  3. 溯源分析                                                 │
│     ├─ 检查访问日志                                         │
│     ├─ 审计日志：/var/log/audit/audit.log                   │
│     └─ Docker 日志：journalctl -u docker                    │
│                                                             │
│  4. 根因修复                                                 │
│     └─ 确保 Docker Daemon 不监听 TCP 端口                   │
│        （检查 /etc/docker/daemon.json）                      │
└─────────────────────────────────────────────────────────────┘
```

**Claude Code 输出示例：**
```
🔴 CRITICAL: Docker Daemon 公网暴露

这是最危险的安全漏洞！攻击者可能已获得服务器控制权。

已完成紧急阻断：
- [x] iptables 规则已添加

⚠️ Runbook 指示后续操作：
1. 检查是否有恶意容器/镜像
2. 审计近期 Docker 操作记录
3. 检查 crontab 是否被植入后门
4. 修改 Docker Daemon 配置

建议立即执行入侵评估！是否继续？
```

#### 事件3：Redis/MongoDB 公网暴露 + 无认证

**触发条件：** Redis(6379/6380) 或 MongoDB(27017) 可公网访问且无密码

**风险等级：** 严重 - 数据泄露/挖矿木马/SSH公钥植入

**响应流程：**

```
┌─────────────────────────────────────────────────────────────┐
│  1. 立即封堵                                                 │
│     ├─ 重启容器绑定 127.0.0.1                               │
│     └─ 或：iptables 临时阻断                                │
│                                                             │
│  2. 入侵检查                                                 │
│  ├─ Redis:                                                  │
│  │   ├─ 检查 keys 数量异常                                  │
│  │   ├─ 检查是否有 SSH 公钥植入（KEYS *）                    │
│  │   └─ 检查 crontab 是否被篡改                             │
│  ├─ MongoDB:                                                │
│  │   ├─ 检查数据库列表                                      │
│  │   └─ 检查 admin.system.users 是否有异常账户              │
│                                                             │
│  3. 数据验证                                                 │
│     ├─ 对比数据量是否异常                                    │
│     └─ 检查敏感数据是否泄露                                  │
│                                                             │
│  4. 根因修复                                                 │
│     ├─ 设置强密码                                           │
│     ├─ 绑定 127.0.0.1                                       │
│     └─ 更新应用配置中的连接字符串                            │
│                                                             │
│  5. 关联服务重启                                             │
│     └─ 重启依赖该数据库的微服务                             │
└─────────────────────────────────────────────────────────────┘
```

**Claude Code 输出示例：**
```
🔴 CRITICAL: Redis 公网暴露且无密码

发现 6380 端口可公网访问，Redis 无密码保护。

已完成端口封堵：
- [x] 容器已重启，绑定 127.0.0.1

⚠️ Runbook 指示后续操作：
1. 检查 Redis 是否被植入 SSH 公钥
2. 检查 crontab 是否被篡改
3. 设置 Redis 密码
4. 更新应用配置
5. 重启依赖 Redis 的微服务

请确认是否执行入侵检查？
```

### 🟠 WARNING（24小时内处理）

#### 事件4：Nginx 版本暴露

**触发条件：** Server 头显示 Nginx 版本号

**修复：** `/etc/nginx/nginx.conf` 添加 `server_tokens off;`

---

#### 事件5：无限流配置

**触发条件：** 未检测到 `limit_req` 配置

**风险：** CC攻击、暴力破解

**修复：**
```nginx
limit_req_zone $binary_remote_addr zone=api:10m rate=10r/s;
location /api/ {
    limit_req zone=api burst=20 nodelay;
}
```

---

#### 事件6：容器无资源限制

**触发条件：** 容器未设置 Memory/CPU 限制

**风险：** 资源耗尽影响其他服务

**修复：**
```bash
docker update --memory="1g" --cpus="1" <container>
```

---

#### 事件7：源站 IP 暴露

**触发条件：** 域名使用 CDN，但源站 IP 可直接访问

**修复：**
1. 配置 Nginx 仅允许 CDN IP 访问
2. 或使用 Tailscale 零信任网络

---

### 🟢 INFO（仅记录）

#### 事件8：SSH 暴力破解尝试

**检测：** `/var/log/auth.log` 中大量 Failed password

**建议：**
- 启用 fail2ban
- 或使用 Tailscale 隐藏 SSH 端口

## 修复后验证清单

每个事件修复后，必须执行验证：

```bash
# 1. 端口绑定验证
netstat -tlnp | grep -E '9200|6379|27017|2375'
# 期望：全部显示 127.0.0.1:端口

# 2. 外部可达性测试
nmap -p 9200,6379,27017 <公网IP>
# 期望：全部显示 closed 或 filtered

# 3. 在线验证
# https://www.yougetsignal.com/tools/open-ports/

# 4. 服务可用性验证
curl http://127.0.0.1:9200/_cluster/health
redis-cli -a <password> ping
```
